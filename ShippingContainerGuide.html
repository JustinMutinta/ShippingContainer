1 What the App Does

The Shipping Container Manager simulates loading cargo onto a ship. The ship carries five storage units — three standard containers (each with five item slots) and two flatbed platforms (one slot each, for large vehicles). A pool of items (boxes, vehicles, crates) can be loaded into and unloaded from those containers through a graphical interface.

Ship ├── Container 1 [ slot 0 ][ slot 1 ][ slot 2 ][ slot 3 ][ slot 4 ] ├── Container 2 [ slot 0 ][ slot 1 ][ slot 2 ][ slot 3 ][ slot 4 ] ├── Container 3 [ slot 0 ][ slot 1 ][ slot 2 ][ slot 3 ][ slot 4 ] ├── Platform 1 [ slot 0 ] └── Platform 2 [ slot 0 ]

There are three windows in the app:

  • Main Menu — the launcher that opens the other two windows.
  • Add / Remove — move items between the available pool and a container.
  • View Contents — inspect any container in a sortable table.

3 Core Concept: The Container Class

A single class, Container, represents every object in the app — both the items being shipped (boxes, vehicles) and the shipping containers themselves. This works because they share the same properties: dimensions, weight, a description, and a serial number.

Fields

public class Container {
    public int    height, width, length, weight;
    public String description;
    public String serialNumber;
    ...
}

Fields are declared public so that other classes can read and write them directly (e.g. item.description). In a larger application you would use private fields with getters and setters, but public is fine for a project this size.

Multiple Constructors (Overloading)

The class has four constructors, each designed for a different kind of object. Java picks the right one based on what arguments you pass — this is called constructor overloading.

ConstructorUsed for
Container()Empty object — all fields are null/zero.
Container(description)A small box: fixed 10×10×10, 5 lbs, no serial number.
Container(description, serialNumber)A 20-foot shipping container: fixed 10×10×20, 4 800 lbs.
Container(h, w, l, wt, desc, serial)Custom object — vehicles, large freight, etc.
// Creates a small box — only needs a name
Container box1 = new Container("Box 1");

// Creates a vehicle with exact dimensions
Container bmw = new Container(60, 60, 120, 450, "BMW Vehicle", "45RFT90RV");

toString()

Overriding toString() makes the object display its description automatically wherever Java needs to convert it to text — most importantly, inside a JList. 

@Override
public String toString() {
    return description;
}

4 Shared Data: The ShipData Class

Both the Add/Remove window and the View Contents window need to see the same cargo. If each window stored its own copy of the containers, changes made in one window would not appear in the other. The solution is a shared data model — one ShipData object that both windows reference.

Design pattern: shared model
MainWindows creates one ShipData instance and passes it into both child windows via their constructors. Because Java passes objects by reference, every window looks at the exact same data in memory.

What ShipData Owns

public class ShipData {
    public Container[]          container1 = new Container[5];
    public Container[]          container2 = new Container[5];
    public Container[]          container3 = new Container[5];
    public Container[]          platform1  = new Container[1];
    public Container[]          platform2  = new Container[1];
    public ArrayList<Container> availableItems = new ArrayList<>();
}

Each container is a fixed-length array of Container objects. An empty slot is simply null. The available items live in an ArrayList because the list grows and shrinks as items are moved.

Array vs ArrayList

Empty slotnullItem is simply removed

Add and Remove Logic

ShipData centralises the add/remove logic so neither window has to implement it independently: 

// Finds the first empty slot and places the item there.
public boolean addToContainer(Container item, String containerName) {
    Container[] target = getContainerByName(containerName);
    if (target == null) return false;
    for (int i = 0; i < target.length; i++) {
        if (target[i] == null) {
            target[i] = item;
            availableItems.remove(item);
            return true;
        }
    }
    return false; // all slots full
}
Why return boolean?
Returning true or false lets the calling window know whether the operation worked — so it can show the user a helpful status message rather than silently doing nothing.

5 Starting the App: Main.java

public class Main {
    public static void main(String[] args) {
        try {
            UIManager.setLookAndFeel("javax.swing.plaf.nimbus.NimbusLookAndFeel");
        } catch (Exception e) { }

        SwingUtilities.invokeLater(MainWindows::new);
    }
}

Look and Feel

Java Swing ships with several visual themes called Look and Feels. The default looks quite dated. Nimbus is a modern built-in theme that requires no extra libraries. UIManager.setLookAndFeel() must be called before any windows are created, otherwise they use the default theme.

SwingUtilities.invokeLater

Swing is not thread-safe. All GUI code must run on a special thread called the Event Dispatch Thread (EDT). invokeLater() schedules the window creation to run on that thread. MainWindows::new is a constructor reference — shorthand for () -> new MainWindows().

Common mistake
Calling new MainWindows() directly in main() (without invokeLater) works most of the time but is technically incorrect and can cause subtle bugs on some systems.

Lambda Event Listeners

When a button is clicked, Swing fires an ActionEvent. You attach a listener to respond to it. Modern Java uses a lambda expression — a short anonymous function: 

// Old style (anonymous class — verbose)
button.addActionListener(new ActionListener() {
    public void actionPerformed(ActionEvent e) {
        new AddRemoveWindow(data);
    }
});

// New style (lambda — concise)
button.addActionListener(e -> new AddRemoveWindow(data));

7 Adding and Removing Items: AddRemoveWindow.java

This is the most complex window. It shows two lists side by side and lets the user move items between them.

┌─────────────────────────────────────────────────────────────┐ │ Selected container: [ Container 1 ▼ ] │ ├─────────────────────┬───────────┬───────────────────────────┤ │ Available Items │ │ Container 1 (2/5 used) │ │ ───────────────── │ Add >> │ ───────────────────── │ │ Box 1 │ │ Box 3 │ │ Box 2 │ << Remove │ Honda Vehicle │ │ Box 4 │ │ │ │ BMW Vehicle │ │ │ │ ... │ │ │ ├─────────────────────┴───────────┴───────────────────────────┤ │ Added "Box 3" to Container 1. │ └─────────────────────────────────────────────────────────────┘

JList and DefaultListModel

A JList displays a scrollable list of items. The data it shows comes from a model — a DefaultListModel. Separating the display from the data means you only need to change the model; the list redraws itself automatically. 

// The model holds the data
DefaultListModel<Container> availableModel = new DefaultListModel<>();

// The JList displays whatever is in the model
JList<Container> availableList = new JList<>(availableModel);

// Updating is simple — the list redraws automatically
availableModel.clear();
for (Container c : data.availableItems) {
    availableModel.addElement(c);  // calls c.toString() to display
}

JScrollPane

A JScrollPane wraps any component and adds scroll bars when the content is taller than the visible area. Without it, items that overflow the list would simply be hidden with no way to reach them. 

leftPanel.add(new JScrollPane(availableList), BorderLayout.CENTER);

BorderLayout

BorderLayout divides a panel into five named regions. The center region takes all remaining space.

┌──────────────── NORTH ────────────────┐ │ WEST │ CENTER │ EAST │ └──────────────── SOUTH ────────────────┘

In AddRemoveWindow, the top combo box goes to NORTH, the three-panel layout (lists + buttons) fills CENTER, and the status label sits in SOUTH.

The Add Logic (step by step)

  1. The user clicks a row in the left list → availableList.getSelectedValue() returns that Container.
  2. The user selects a container from the combo box → containerComboBox.getSelectedItem() returns its name.
  3. data.addToContainer(item, containerName) is called — it finds the first null slot, assigns the item, and removes it from availableItems.
  4. Both lists are refreshed by clearing their models and re-adding from data.
  5. The status label shows a confirmation or an error message.

TitledBorder — the right way to update it

The right panel has a border that shows the slot count. To update that text, the border object is stored as a class field so it can be accessed directly: 

// Stored as a field
private TitledBorder containerBorder;

// Created once in buildContentPanel()
containerBorder = BorderFactory.createTitledBorder("Container Contents");
rightPanel.setBorder(containerBorder);

// Updated later without touching the component tree
containerBorder.setTitle(selected + "  (2/5 slots used)");
containerList.getParent().repaint();
Common pitfall: component tree traversal
An earlier version tried to reach the panel via containerList.getParent().getParent() and cast it to JPanel. This caused a ClassCastException at runtime because getParent() on a JList inside a JScrollPane returns a JViewport, and the next parent is the JScrollPane — not the JPanel. Always store UI objects you need to update as class fields rather than navigating the component tree.

8 Viewing Contents: ViewContentsWindow.java

This window shows the contents of any container in a sortable table.

JTable and DefaultTableModel

Just like JList uses DefaultListModel, a JTable uses a DefaultTableModel that holds rows and columns. Defining the columns up front gives the table its headers. 

String[] columns = {"Description", "Serial Number",
                     "Height", "Width", "Length", "Weight (lbs)"};

DefaultTableModel tableModel = new DefaultTableModel(columns, 0) {
    @Override
    public boolean isCellEditable(int row, int col) {
        return false; // prevent the user from typing in the table
    }
};

Populating the table is done by adding one row per item:

tableModel.setRowCount(0); // clear old rows
for (Container c : slots) {
    if (c != null) {
        tableModel.addRow(new Object[]{
            c.description, c.serialNumber,
            c.height, c.width, c.length, c.weight
        });
    }
}

Why a Refresh Button?

The View Contents window is a separate window from Add/Remove. When the user adds an item in Add/Remove and then looks at View Contents, the table does not update automatically — it only re-reads ShipData when the user interacts with it. The Refresh button is an explicit way to trigger that re-read.

Alternative: Observer / Event pattern
In larger applications, you would implement the Observer pattern: ShipData would maintain a list of listeners, and any window that cares about data changes would register itself. When data changes, ShipData notifies all listeners to refresh. This keeps windows in sync automatically without a manual Refresh button.

9 Key Java & Swing Concepts Explained

Inheritance (extends)

Every window class extends JFrame. That means it automatically has a title bar, close button, resize handles, and the ability to contain components — all without writing any of that code yourself. 

public class MainWindows extends JFrame { ... }

Generics (<T>)

DefaultListModel<Container> means the model only holds Container objects. Generics let the compiler catch type mistakes before the program runs, rather than crashing at runtime with a ClassCastException.

final Fields

Fields declared final can only be assigned once. Using final for the list models and JLists makes it clear they will never be replaced — only their contents will change.

Layout Managers

LayoutBehaviourUsed in
BorderLayoutFive named regions; CENTER expands to fill space.Main frame of each window
BoxLayout (Y_AXIS)Stacks components top to bottom.Main menu buttons, Add/Remove center panel
FlowLayoutPlaces components left to right, wrapping when full.Top bar (label + combo box)

DISPOSE_ON_CLOSE vs EXIT_ON_CLOSE

The main window uses EXIT_ON_CLOSE — closing it ends the whole program. The child windows use DISPOSE_ON_CLOSE — closing them only releases their memory, leaving the main window (and all data) alive.

setLocationRelativeTo(null)

Passing null centres the window on screen. Calling this after pack() or after setting the size ensures the window size is known before the position is calculated.

10 How the Pieces Connect

Main.java │ Sets Nimbus look-and-feel │ Calls SwingUtilities.invokeLater(MainWindows::new) │ └─▶ MainWindows │ Creates ONE ShipData instance │ Shows three buttons │ ├─▶ AddRemoveWindow(data) │ │ Receives the shared ShipData reference │ │ Left JList ← data.availableItems │ │ Right JList ← selected container array │ │ Add button → data.addToContainer(item, name) │ └─ Remove btn → data.removeFromContainer(item, name) │ └─▶ ViewContentsWindow(data) │ Receives the same ShipData reference │ JTable ← selected container array └─ Refresh button → re-reads data and redraws table

Why one ShipData instance matters

Because both child windows hold a reference to the same ShipData object, an item added in AddRemoveWindow is immediately visible when ViewContentsWindow refreshes. No copying, no synchronisation — they both point at the same memory.

What to build next
  • Persistence — save and load the ship's state using a file or an SQLite database (via the JDBC driver) so cargo survives app restarts.
  • Observer pattern — have ShipData automatically notify open windows when data changes, removing the need for a manual Refresh button.
  • Input validation — add a form to create new items rather than having them hardcoded in ShipData.
  • Weight limits — prevent overloading a container by checking the combined weight of its contents against a maximum before adding an item.
